# Phase 2 Handoff: Worker System (Health Checker)

เอกสารฉบับนี้จัดทำขึ้นสำหรับ Developer หรือทีมงานที่จะเข้ามารับช่วงต่อในการพัฒนาระบบเช็คสถานะเพจอัตโนมัติ (Worker) ใน **Phase 2 (Database & Telegram Integration)**

## 🎯 สรุปเป้าหมายสูงสุดของ Worker (The End Goal)
ระบบนี้จะต้องเป็น **Background Service (Daemon)** ที่ทำงานแยกอิสระจาก Next.js
* ทำหน้าที่: ดึงรายชื่อ "เพจที่ยัง Active อยู่" จาก Database กลาง 
* เพื่อ: เปิดเบราว์เซอร์ (ด้วย Puppeteer) วิ่งเข้าไปเช็คหน้า Facebook จริงตามเวลาที่ตั้งไว้ (12:00, 16:00, 19:00, 23:00)
* เมื่อพบความผิดปกติ (เพจปลิว/ถูกปิดกั้น): ระบบจะต้อง **1) อัปเดตสถานะกลับไปที่ Database โดนอัตโนมัติ** และ **2) ส่งข้อความแจ้งเตือนเข้ากลุ่ม Telegram ทันที**

---

## 🛠️ โครงสร้างปัจจุบัน (ณ ตอนส่งมอบเครื่องมือ)
* โค้ดอยู่ในโฟลเดอร์ `worker/`
* `health-checker.js`: ใช้ `puppeteer` เช็คสถานะได้ 100% แม่นยำ และใช้ `node-cron` ตั้งเวลาทำงานเรียบร้อย
* **ข้อจำกัดปัจจุบัน:** ตอนนี้ระบบอ่านรายชื่อเพจเป้าหมายจากไฟล์ `config.json` และโชว์ผลลัพธ์แค่ทาง Console (เพราะตั้งใจให้เป็น Phase 1 ก่อนต่อ DB)

---

## 🚀 แผนงานที่ต้องทำต่อ (Phase 2 Action Plan)

### Step 1: เชื่อมต่อ Database (Fetch & Update)
เมื่อโปรเจกต์เปิดใช้งาน Database บริหารจัดการ (เช่น Supabase หรือ PostgreSQL) เต็มรูปแบบแล้ว ทีมงานต้อง:
1. ติดตั้งไลบรารี DB ใน Worker (รัน `npm install @supabase/supabase-js` ในโฟลเดอร์ worker)
2. นำ Connection Strings ไปใส่ตั้งค่าผ่าน `.env`
3. เข้าไปแก้โค้ดใน `health-checker.js` 
   * **จุดที่ต้องลบ:** ลบโค้ดอ่านไฟล์ `config.json` ทิ้ง
   * **จุดที่ต้องเพิ่ม (Fetch Input):** เขียนคำสั่ง Select ข้อมูลจากตาราง `Pages` ที่มีเงื่อนไข `status = 'Active'` มาใส่ในตัวแปร `pages`
   * **จุดที่ต้องเพิ่ม (Scraping Data - สำคัญมาก):** อัปเกรดโค้ด Puppeteer ให้ทำการดึง **"รูปโปรไฟล์ (Profile Picture URL)"** และ **"ยอดผู้ติดตาม (Followers Count)"** กลับมาด้วย เผื่อเอาไว้โชว์บน Dashboard ให้แอดมินดูแบบไม่หน่วงเว็บ
   * **จุดที่ต้องเพิ่ม (Update Output):** 
     - นำยอด Followers และ URL รูปโปรไฟล์ที่ดูดมาได้ อัปเดตทับลงตาราง `Pages` (เพื่อให้ Dashboard ดึงไปใช้ต่อ)
     - และถ้าเงื่อนไขเช็คว่าเพจไหนพัง (`status === 'broken'`) ให้ Update DB ปรับสถานะเป็น `Problem` ทันที

### Step 2: เชื่อมต่อ Telegram Notifications
1. แอดมินต้องไปสร้างบอทกับ **BotFather** ในแอพ Telegram ก่อน
2. นำ **Bot Token** และ **Chat ID** ของกลุ่มออฟฟิศ มาใส่ใน `.env` ของ Worker
3. ติดตั้งไลบรารี `npm install axios` 
4. เข้าไปแก้โค้ดใน `health-checker.js` ตรงช่วงประเมินผลสรุปท้ายสุด:
   * ให้นำรายชื่อเพจตัวที่บิน (Broken Array) มาจัดเรียงเป็น String ข้อความสวยๆ
   * ใช้ `axios.post` ยิงข้อความนั้นไปหา Telegram API
   * _ตัวอย่าง Payload:_ `https://api.telegram.org/bot<TOKEN>/sendMessage?chat_id=<CHAT_ID>&text=<MESSAGE>`

## 💡 คำแนะนำสำหรับการนำขึ้น Production
เมื่อทำ Step 1 และ Step 2 เสร็จ การนำสคริปต์นี้ไปรันใช้งานจริง (Production) มี 2 ทางเลือก:

**🔴 ทางเลือกที่ 1: เช่า Cloud Server (มีต้นทุนประมาณ 200-300 บาท/เดือน)**
* เหมาะสำหรับทีมที่ไม่มีคอมพิวเตอร์ออฟฟิศเปิดทิ้งไว้ 24 ชม.
* ควรเช่า Server เล็กๆ (แนะนำ 1GB-2GB RAM) เช่น AWS EC2 (t3.small) หรือ DigitalOcean
* รันและควบคุม Process ด้วยคำสั่ง `pm2 start health-checker.js --name "fb-health-worker"` 

**🟢 ทางเลือกที่ 2: รันบนคอมพิวเตอร์ที่ออฟฟิศ (ต้นทุน 0 บาท - แนะนำ!)**
* เหมาะสำหรับทีมที่มี PC หรือ Mac Mini เสียบปลั๊กเปิดทิ้งไว้ที่ออฟฟิศ 24 ชม.
* **ข้อดีสุดยอด:** อินเทอร์เน็ตออฟฟิศ/บ้าน (Residential IP) จะไม่ถูก Facebook มองว่าเป็นบอท ทำให้โอกาสโดนบล็อกแทบจะเป็น 0% แตกต่างจาก IP ของ Cloud Server ที่มักจะโดนเพ่งเล็ง
* วิธีการ: แพ็คตัว Worker นี้เป็นไฟล์โปรแกรม `.exe` หรือใช้ PM2 ฝังในเครื่องออฟฟิศให้รันตอนเปิดเครื่องอัตโนมัติ

> **คำเตือนที่สำคัญมาก:** ไม่รัน Worker ตัวนี้บน Vercel หรือ Server เดียวกับที่ใช้รัน Frontend (Next.js) เด็ดขาด เนื่องจาก Puppeteer กินพื้นที่ Memory และ CPU เวลารันเบราว์เซอร์ ซึ่งอาจทำให้ฝั่งเว็บไซต์หลักล่มได้ครับ
